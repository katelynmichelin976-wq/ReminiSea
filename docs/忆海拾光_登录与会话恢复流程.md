## 一、关键全局变量

| 变量 | 初始值 | 含义 |
|------|--------|------|
| `_sb` | `null` | Supabase 客户端实例，initCloud 后创建 |
| `_syncEnabled` | `false` | 云同步是否可用（在线且 token 有效） |
| `_cloudUserEmail` | `''` | 当前登录用户 email，空字符串表示未登录 |
| `_cloudUserId` | `''` | 当前登录用户 UUID |

### localStorage 关键键

| 键名 | 写入时机 | 清除时机 |
|------|----------|----------|
| `yihai_session_backup` | 登录成功、token 刷新、离线恢复 | 登出 |
| `sb-{project}-auth-token` | Supabase SDK 自动管理 | Supabase SDK 自动管理 |
| `yihai_last_cloud_email` | 登录成功 | 不清除（用于下次预填） |

## 二、App 启动流程

```
App 启动（页面加载）
  │
  ├─ [立即] initUI()
  │    └─ 恢复牌组、渲染首页（不依赖云服务）
  │
  └─ [100ms 后] _tryInitCloud() 开始轮询
       │  每 200ms 检查 supabase 对象是否就绪
       │  最多 50 次（约 10 秒）
       │
       ├─ SDK 就绪 → initCloud()
       │    ├─ 注册 offline 事件监听（永久）
       │    ├─ await restoreSession()
       │    ├─ 注册 onAuthStateChange 监听
       │    └─ upsertDeviceRegistry()
       │
       └─ 50 次用完仍未就绪 → 静默放弃 ⚠️ 无任何 UI 反馈
```

## 三、会话恢复流程（restoreSession）

**触发时机**：initCloud 执行时、网络恢复时（_onOnlineRetry）

```
restoreSession()
  │
  ├─ 创建 _sb 客户端
  ├─ 读取本地 token
  │    优先：sb-{project}-auth-token（SDK 原生键）
  │    兜底：yihai_session_backup
  │
  ├─ [路径 A] token 不存在或无 email
  │    → 直接 return
  │    ⚠️ 不调 updateCloudTabUI()
  │    ⚠️ 不注册 online 监听
  │
  └─ token 存在 → Promise.race(getSession(), 7秒超时)
       │
       ├─ [路径 B] 成功（session 有效）
       │    _syncEnabled = true
       │    _cloudUserEmail = session.user.email
       │    写回 yihai_session_backup
       │    → updateCloudTabUI() ✅
       │    ⚠️ 不注册 online 监听
       │
       ├─ [路径 C] 失败（超时 or 无 session）
       │    _syncEnabled = false
       │    _cloudUserEmail = parsed.user.email
       │    → updateCloudTabUI() ✅
       │    → 注册 online 监听（一次性）✅
       │
       └─ [路径 D] 抛异常（catch）
            _syncEnabled = false
            _cloudUserEmail 不赋值（维持空字符串）⚠️
            → updateCloudTabUI() ✅
            ⚠️ 不注册 online 监听
```

## 四、网络状态事件

### offline 事件

**注册时机**：initCloud 执行时，永久存在

```
网络断开
  └─ 若 _syncEnabled 为 true
       → _syncEnabled = false
       → updateCloudTabUI()
```

### online 事件（_onOnlineRetry）

**注册时机**：仅 restoreSession 路径 C（getSession 失败/超时）注册，**一次性**

```
网络恢复
  ├─ 注销自身监听（保证只触发一次）
  ├─ 重新跑 restoreSession()
  └─ 若 _syncEnabled = true → runSync()
```

**路径 A / D 不注册 online 监听**，网络恢复后不会自动重试。

### online 事件的底层机制

`window.addEventListener('online', handler)` 是浏览器原生事件，由操作系统网络接口状态变化触发（WiFi 重连、4G 恢复、飞行模式关闭等），触发时 `navigator.onLine` 同时变为 `true`。代码本身不轮询，完全依赖系统推送。

**online 事件 ≠ Supabase 可达**

系统认为联网（如连上 WiFi），但 `juzkonrzfyvchqxzmlpr.supabase.co` 实际可能还不通（DNS 未解析、Tokyo 节点暂时不可达）。`_onOnlineRetry` 触发后立即跑 `restoreSession`，若 `getSession()` 仍失败（7 秒超时兜底），会再次走路径 C，重新注册 online 监听，等下次网络事件再试。

行为是「联网就重试，重试失败继续等」，不是「联网即成功」。

## 五、账号页渲染（renderAccount）

**触发时机**：updateCloudTabUI()、showAccount()、doAccountLogout()

```
renderAccount()
  │
  ├─ 条件：!_cloudUserEmail && localStorage.getItem('yihai_session_backup')
  │    → 显示「正在恢复登录…」
  │    ⚠️ 用 backup 键是否存在代理「恢复中」状态，不是显式标志
  │
  ├─ 条件：_syncEnabled && _cloudUserEmail
  │    → 显示已登录界面
  │
  └─ 其他
       → 显示登录表单（预填 yihai_last_cloud_email）
```

## 六、手动登录流程（doAccountLogin）

**触发时机**：用户点击「登录」按钮

```
doAccountLogin()
  │
  ├─ 校验输入（email / 密码非空）
  ├─ 按钮 disabled，文字改为「登录中…」
  │
  ├─ [try]
  │    ├─ 创建 _sb 客户端
  │    ├─ await signInWithPassword()  ⚠️ 无超时，网络挂住时永远等待
  │    ├─ 登录成功：填充全局变量，写 yihai_session_backup
  │    ├─ await migrateDeviceRecordsToUser()
  │    ├─ updateCloudTabUI()
  │    └─ runSync()（不 await，后台执行）
  │
  ├─ [catch]  → 显示错误信息，_syncEnabled = false
  │
  └─ [finally] → 按钮恢复
```

## 七、token 自动刷新（onAuthStateChange）

**注册时机**：initCloud 执行后，_sb 存在时注册，永久存在

```
TOKEN_REFRESHED
  → _cloudUserEmail / _cloudUserId 更新，_syncEnabled = true
  → 写回 yihai_session_backup
  → updateCloudTabUI()

SIGNED_OUT（仅当 _syncEnabled 为 true 时处理）
  → _syncEnabled = false，_cloudUserEmail = ''
  → updateCloudTabUI()
```

## 八、登出流程（doAccountLogout）

```
doAccountLogout()
  ├─ _sb.auth.signOut()
  ├─ _syncEnabled = false，_cloudUserEmail = ''，_sb = null
  ├─ 清除 yihai_session_backup、yihai_global_sync_ts
  └─ 更新所有 UI
```

## 九、已知缺陷

| # | 位置 | 问题 | 触发条件 | 状态 |
|---|------|------|----------|------|
| 1 | `renderAccount` | 用 `yihai_session_backup` 是否存在代理「恢复中」状态 | backup 存在且 `_cloudUserEmail` 为空时卡住，无法自愈 | ✅ 已修复（`_sessionRestoring` 标志） |
| 2 | `restoreSession` 路径 A | 直接 return，不调 `updateCloudTabUI()` | backup 损坏解析为 null 时 UI 不更新 | ✅ 已修复（finally 统一调用） |
| 3 | `restoreSession` 路径 D | `_cloudUserEmail` 不赋值，维持空字符串 | getSession() 抛异常，backup 仍存在时 UI 卡在「恢复中」 | ✅ 已修复（finally 统一调用） |
| 4 | `restoreSession` 路径 D | 不注册 online 监听 | 网络异常导致 catch，网络恢复后不自动重试 | ⏭ 跳过（路径 D 在实践中极少触发） |
| 5 | `_tryInitCloud` 放弃时 | 静默放弃，不更新 UI | SDK CDN 加载失败，backup 存在时 UI 卡在「恢复中」 | ✅ 已修复（`_sessionRestoring` 副作用覆盖） |
| 6 | `doAccountLogin` | `signInWithPassword` 无超时保护 | 网络请求挂起时按钮永久 disabled | ✅ 已修复（15s Promise.race） |

### 缺陷叠加场景（本次实际触发路径）

```
1. 新版本部署，用户点刷新按钮
2. location.reload(true) → 浏览器重新下载所有资源（含 SDK）
3. SDK CDN 加载变慢，_tryInitCloud 轮询等待中
4. 用户打开账号页
   _cloudUserEmail = ''（初始值），backup 仍在
   → renderAccount 显示「正在恢复登录…」

5a. SDK 加载完，restoreSession 跑路径 B/C
    → updateCloudTabUI() → UI 正常推进  ✅

5b. SDK 加载失败，_tryInitCloud 静默放弃（缺陷 5）
    → restoreSession 从未运行，UI 永远卡住  ❌

5c. restoreSession 走路径 D（缺陷 3）
    → _cloudUserEmail 仍空，backup 仍在，UI 卡住
    → 且不注册 online 监听（缺陷 4），网络恢复也无法自愈  ❌
```

## 十一、测试策略

### 结论：使用 Playwright，不用单元测试

登录/恢复流程的缺陷本质是**状态没有被正确推进**，涉及 localStorage、异步时序、DOM 状态、网络事件的协作。单元测试需要 mock localStorage、Supabase SDK、`window.addEventListener`、网络超时，mock 复杂度会超过被测代码本身，且 mock 越多测的越不是真实行为。

Playwright 在真实浏览器里运行，能直接重现本次 bug 的触发路径。

> 单元测试适合项目里的纯函数逻辑，如 SM-2 `processAnswer` 状态机、`buildOptions` 洗牌均匀性验证，这些与浏览器环境无关。

### Playwright 网络模拟能力

```js
// 断网
await context.setOffline(true);

// 网络恢复
await context.setOffline(false);

// 模拟 Supabase 请求超时
await page.route('**/auth/v1/**', route => route.abort('timedout'));

// 模拟 Supabase 返回指定错误
await page.route('**/auth/v1/**', route =>
  route.fulfill({ status: 429, body: 'rate limited' })
);

// 模拟 SDK CDN 加载失败
await page.route('**/supabase.min.js', route => route.abort());

// 模拟 SDK 加载慢（触发 _tryInitCloud 长时间轮询）
await page.route('**/supabase.min.js', async route => {
  await new Promise(r => setTimeout(r, 8000));
  await route.continue();
});
```

这些是浏览器层面的真实模拟，不是 mock——SDK 真的没加载进来，`_tryInitCloud` 真的会跑完 50 次，`online` 事件真的会触发。

### 测试用例覆盖矩阵

| 测试用例 | 覆盖缺陷 |
|----------|----------|
| 首次打开无 backup → 直接显示登录表单 | 路径 A |
| backup 有效 → 自动恢复到已登录状态 | 路径 B |
| 断网打开 → 显示已登录（离线态）→ 联网 → 触发同步 | 路径 C + online 事件 |
| SDK 加载慢，期间打开账号页 → 不卡死，最终推进到正确状态 | 缺陷 1、5 叠加场景 |
| SDK CDN 加载失败（50 次轮询耗尽）→ 显示登录表单，不卡死 | 缺陷 5 |
| getSession() 抛异常 → 不卡死，显示登录表单 | 缺陷 3 |
| 登录请求超时 → 按钮恢复，显示错误提示 | 缺陷 6 |
| 断网后 getSession 失败 → 网络恢复 → 自动重试 restoreSession | 缺陷 4 |
