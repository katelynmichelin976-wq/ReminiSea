# 忆海拾光 — 家庭协作双端设计

**日期**: 2026-05-24
**状态**: 设计已确认，待实施计划

## 1. 产品定位

忆海拾光（Memory Glimmers）—— 为被照护者而生的家庭协作卡片 PWA。

核心价值：**声音跨越距离**。照顾者制作图+文+音的卡片，被照护者在极简界面浏览和聆听。多人贡献，一人使用。

## 2. 目标用户

### 照顾者

- 父母、祖辈、子女等家庭成员，多人协作
- 有正常移动设备使用能力
- 角色：创建卡片、录音、管理内容、邀请其他照顾者

### 被照护者

- 认知能力不足的用户（儿童、老人、特需人士）
- 不会使用微信或其他复杂 App
- 不需要打字、不需要选择、不需要登录概念
- 只需要：看到卡片 → 点一下 → 听到家人的声音

## 3. 双端设计

同一套代码，根据账号角色切换 UI 模式。登录时识别角色 → 加载对应界面。

### 3.1 照顾者端（现有 App，略调整）

| 项目 | 说明 |
|---|---|
| 登录 | 邮箱 + 密码 |
| 界面 | 完整功能：首页（牌组列表）、练习、我的、设置、账户 |
| 功能 | 制卡、录音、管理牌组、云同步、邀请家人、SRS 练习 |
| 权限 | 全部 |

### 3.2 被照护者端（新增，核心设计）

| 项目 | 说明 |
|---|---|
| 登录 | **无感**。首次由照顾者协助登录，之后自动静默恢复。被照护者从始至终不知道"登录"存在 |
| 启动 | 打开 App → 自动恢复 session → 直接进入卡片浏览 |
| 界面 | **极简**。无文字、无菜单、无按钮（除翻页）、无设置、无选择 |
| 交互 | 打开即看到卡片（图片+文字），点一下播声音，左右滑翻页 |
| 内容 | 照顾者端预设的牌组，自动按 SRS 或固定顺序出牌 |
| 权限 | 只读。不显示任何排行榜/评分/统计/管理功能 |

### 3.3 角色判定

账号关联 `user_role` 字段（`caregiver` / `care_receiver`）。登录成功后根据角色：
- `caregiver` → 加载完整 UI
- `care_receiver` → 加载极简 UI

角色由照顾者在 Supabase 后台或账户页面指定。首次创建家庭组时自动指定被照护者角色。

## 4. 无感登录设计

### 4.1 目标

- 首次由照顾者协助登录一次
- 之后每次打开 App 自动恢复，无需任何用户操作
- 和微信一样：打开就在，只有换设备才需要重新验证

### 4.2 实现方案

**方案 A — 存密码自动重登**（推荐，和 Anki 一致）：

```
首次登录：照顾者输入 email + pwd → signIn → 存 session + 存 btoa(email||pwd)
后续启动：读 backup → getSession() + 7s 超时
  → 成功：在线
  → 失败：读 yihai_cred → 静默 signIn → 在线
  → 静默登录也失败 → 显示"需要照顾者协助登录"提示
```

存密码用 `btoa`（Base64 编码，非加密）。在 PWA standalone 环境（无 DevTools 入口）下攻击面极小，可接受。

**方案 B — 长期 Refresh Token**（备选）：

Supabase 默认 refresh token 永不过期。但实际中 localStorage 被清、refresh token 重用检测等会导致意外登出。不如方案 A 可靠。

### 4.3 降级处理

自动恢复完全失败时：
- 被照护者端显示：简单的图示 + "请让家人帮忙重新登录"
- 不暴露密码输入框、不显示技术错误信息

## 5. 家庭协作模型

### 5.1 协作模式

- 多人照顾者 → 一人被照护者
- 自然贡献，无主编分工
- 同一家庭组内卡片和对所有人可见

### 5.2 家庭组

- 一人创建家庭组 → 生成邀请链接/码
- 其他照顾者通过邀请加入
- 被照护者由创建人添加到家庭组

### 5.3 卡片制作与分发

- 照顾者在自己的设备上制卡
- 卡片通过 Supabase 云同步到家庭组共享牌组
- 被照护者端自动拉取，无需手动操作

## 6. 不做

- 不拆分两个独立 App（同一套代码切换模式）
- 不现在做邀请 UI 的前端实现（后台手动绑定优先）
- 被照护者端不做任何编辑/设置/选择/评分
- 不引入 WebCrypto 加密（存密码 btoa 够用）
