# 忆海拾光 · v4.11 → v5 UI 迁移设计

> 日期：2026-05-22
> 状态：设计已确认，待进入实现计划（writing-plans）
> 关联：`yihai_v4.11.html`（生产代码）、`yihai_design_v5.1.html`（UI 原型）、`index_v49.html`（制卡工具，已暂停）

---

## 一、背景与动机

v4.11 是线上试点版本，AD 患者（妈妈账号）正在使用。v5 设计了新的双模式 UI 交互，并计划融合制卡功能（原 `index_v49.html`）。

**核心驱动力**：未来要持续推进新卡牌类型。如果 v4.11 和 v5 设计稿各自演进，将来合并时**测试成本和迁移成本会累积**。需要选一条能控制这种成本的演进路径。

**关键约束**：
1. v4.11 在线试点不能中断（AD 患者在用，且界面频繁变动对 AD 患者有害）
2. 新卡牌类型 = 答题 UI 扩展，**SRS 逻辑不变**

---

## 二、路径决策：以 v4.11 为基础演进（路径 A）

在三条候选路径中选定**路径 A：以 v4.11 为基础，逐步迁移 UI**。

| 路径 | 描述 | 结论 |
|------|------|------|
| **A（选定）** | v4.11 为基座 → 应用 v5 UI → 加新卡片类型 → 制卡 → Capacitor | 单一代码库持续演进，测试始终绿，无大爆炸式合并 |
| B | v5 设计稿为基座 → 回填 v4.11 逻辑（SRS/IDB/sync） | 271KB 战场测试代码迁移风险集中；并行维护两套代码 = 正是要避免的分叉成本 |
| C | 先加制卡再迁 UI | 制卡 UI 在旧框架里加，迁 v5 时需二次调整 |

**选 A 的核心理由**：用户担忧的是"分叉越大测试成本越高"，而路径 A 根本不存在分叉 —— v4.11 本身就在向 v5 演进。路径 B 反而把并行期最大化。

**重要澄清**：路径 A 是**代码策略**（单一代码库演进、测试常绿），不等于"每个中间版本都发给患者"。发布节奏见第四节。

## 三、i18n 多语言模型（前期规划）

App 在美区上架，国内外用户都可用。**不是"一张卡支持多语言"，而是支持不同语言的内容，且一张卡的不同字段可以是不同语言。**

### 两类语言，各自独立

| 类别 | 跟随什么 | 例子（西班牙用户学中文） |
|------|---------|----------------------|
| **界面 chrome + 引导语音** | UI 语言（设备语言，可切换） | 按钮、菜单、提示文字和引导 TTS 都是西班牙语 |
| **卡片字段内容 + 字段 TTS** | 每个字段自己的语言标签 | 名称="苹果"（中文，TTS 中文）；描述="manzana"（西班牙语，TTS 西班牙语） |

### 卡片字段数据模型

```
现在:  card.name = "苹果"
改为:  card.name = { text: "苹果",   lang: "zh-CN" }   → TTS 中文语音
       card.desc = { text: "manzana", lang: "es" }      → TTS 西班牙语语音
```

- **不是翻译副本**（一个字段只有一个值），而是**不同字段可以是不同语言**
- 支持外语学习卡：正面母语、名称目标语、名称 TTS 目标语
- 没有 locale map、没有翻译重复、没有显示回退逻辑

### 字段语言的操作：自动识别 + 偶尔纠正

绝大多数卡**无需手动选语言**：

- **按书写系统（script）自动识别**：Unicode 字符范围判断脚本，零依赖、近 100% 准。"苹果"→CJK→zh，"manzana"→拉丁，"яблоко"→西里尔→ru。
- **可靠覆盖对 TTS 最重要的区分**（CJK vs 西方语音）；把中文用西方语音念这种最糟错误能稳定拦住。
- **同脚本内细分（es vs en vs pt）判不准** → 回退到牌组主语言 / UI 语言。不引入语言检测库（短文本不准 + 体积负担）。

操作模型：
```
1. 建牌组时设"主语言"一次（默认 = 界面语言）
2. 每个字段：脚本与主语言一致 → 继承主语言；脚本不同 → 自动按检测脚本打标
3. 猜错时用户手动改某字段（罕见）
```

效果：纯母语牌组零操作；混合语言学习卡（西班牙语正面 + 中文名称）也零操作（中文字段自动识别）。

## 四、分阶段路线与版本规划

```
主线（PWA）
  阶段 0  地基（先于 Wave 1）
    ├─ 0b  界面 i18n（字符串表 + t() + 设备检测 + 引导语音按 UI 语言）
    └─ 0c  卡片字段语言标签 {text, lang}（.yhspack 带标签，存量迁移 zh-CN）
  阶段 1  Wave 1 单模式 v5 重构（dev.1–dev.5）
  阶段 2  制卡功能（融合 index_v49）→ 发布 v5.0.0（PWA）
  阶段 3  Wave 2 患者模式     → v5.1（PWA）

  （新卡片类型 = 阶段 2 之后在制卡基础上叠加的增量，每个类型独立 mini-design）

─────────────────────────────────────────
独立轨道（PWA 稳定后启动，与主线解耦）
  Capacitor 打包
    ├─ SDK 本地化（Supabase SDK CDN → 本地）
    ├─ TTS 设备级验证（iOS/Android WebView 多语言语音）
    ├─ 相机插件（制卡拍照）
    ├─ 邮件确认 deep link
    └─ 上架（教育类 / 无疗效声明 / 隐私标签）
                                          → v5.2
```

### 版本规划

| 阶段 | 版本 | 发布对象 | 含义 |
|------|------|---------|------|
| 阶段 1 Wave 1 | **v5.0.0-dev** | 不推给患者，内部/测试机验证 | 新 UI 成形期，代码持续演进、Playwright 全程把关 |
| 阶段 2 | **v5.0.0 正式版** | 一次性交付患者 | 新 UI + 制卡功能（v4.11 不具备的新能力）= 真正的 v5.0 |
| 阶段 3 Wave 2 | **v5.1** | 升级 | 患者极简模式 + 模式切换 |
| Capacitor 轨道 | **v5.2** | 上架 | 独立项目，PWA 稳定后启动 |

**关键决策**：
- **v5.0.0 = 阶段 2**（新 UI + 制卡一起），这才是"v4.11 没有的新产品形态"
- 患者从 v4.11 **直接跳到完整的 v5.0**，避免迁移过程中的 UI 抖动（对 AD 患者尤其重要）
- **Capacitor 完全解耦**为独立轨道，等 PWA（v5.1）稳定后再启动，互不阻塞
- SDK 本地化随 Capacitor 轨道（PWA 已有 CDN 离线兜底，主线不需要提前做）

## 五、Wave 1 逐版本拆分

**原则：骨架先行（最高风险先稳定），再逐屏换皮，每版测试绿、功能零丢失。**

v4.11 实际结构：4 个 screen（home/quiz/finish/stats）+ 带标签页的设置浮层（含云端 tab 4）；浏览是 `startBrowse()` 模式；首页底部"浏览/开始"按钮 + 齿轮进设置 + 统计入口。

| 版本 | 内容 | 映射 v4.11 | 验证点 |
|------|------|-----------|--------|
| **dev.1 导航骨架** | 三 Tab（首页/▶练习FAB/我的）替换底部按钮+齿轮+统计入口；各屏内部不动，只重接入口 | 重接 `goHome`/`startSession`/`openStats`/设置浮层 | 所有流程经新导航可达；Playwright 导航断言更新 |
| **dev.2 首页+浏览屏** | 牌组列表→v5 行（名称+张数+语言徽标）；浏览模式→v5 浏览屏（大图+名称TTS+分隔+描述+上下张） | `screen-home` + `startBrowse` | 浏览流程视觉 + 按字段语言选 TTS；Playwright 浏览断言 |
| **dev.3 练习屏** | v5 答题（确认→答案面板→倒计时环继续）；三色计数器（新/学习/复习） | `screen-quiz` + `screen-finish` | SRS 逻辑不动（单测绿）；**Dev Rule 18：跑 Playwright 验 trials≥20** |
| **dev.4 统计屏** | v5 四 Tab（今日/牌组/卡片/记录），补全原型缺口 | `screen-stats` | 数据准确性对照 v4.11 |
| **dev.5 我的页整合** | 设置 + 云端登录/同步 + 语言切换（阶段 0b）整合进"我的" | `settings-overlay`（含云端 tab） | 登录/登出/同步/设置持久化；Playwright 云端断言 |

### 排序理由
- **骨架先行**：三 Tab 是新框架，所有屏都要嵌进去，最先做、最先稳定
- **内容屏由简到繁**：浏览（只读）→ 答题（复杂、近 SRS）→ 统计（只读）→ 我的（整合多块）

## 六、测试策略

- 每版是 UI 改动 → 按 **Dev Rule 8** 先写/更新 Playwright 复现再改
- 现有 Playwright 断言绑 DOM 选择器，会随 UI 变 → **每版更新对应套件的选择器 + 新增 v5 元素断言**
- SRS/v4.4/v4.8/v4.9 单测**全程不应变**（逻辑没动）—— 若单测红了说明误伤逻辑，立即查
- TTS 验证分两层：阶段 0b 验证目标**浏览器**语音可用性（PWA）；设备级 WebView 语音验证归 Capacitor 轨道

---

## 七、风险与缓解

| 风险 | 缓解 |
|------|------|
| UI 大改打乱 Playwright 选择器，回归被误判为"选择器过时" | Dev Rule 8 纪律：先为新行为写断言再改代码 |
| 多语言 TTS 设备语音缺失（尤其 iOS WebView） | 阶段 0b 做字段 TTS 时就先验证目标浏览器语音；设备级验证不拖到 Capacitor 末期 |
| 卡片字段语言模型迁移存量牌组 | 存量字段 → `zh-CN` 必须有迁移测试 |
| .yhspack 格式升级向后兼容 | 旧文件导入测试；格式设计为可扩展（新卡片类型不需再改格式） |
| 新卡片类型未定 | 阶段 2 只做制卡功能（已明确）；新类型作为阶段 2 之后的增量，每个独立 mini-design，不阻塞阶段 2 |

## 八、App Store 合规要点（Capacitor 轨道）

- **类别：教育类**，不以医疗/健康类别上架（对标 Anki）。不触发指南 1.4.1（医疗）、5.1.3（健康数据）
- **真正的红线是功效声明**：不得声称"治疗/延缓阿尔茨海默症""提升记忆力"（FTC 曾处罚此类）。描述机制（"家庭相册 + 间隔重复记忆练习"），不承诺医学结果
- 不接 HealthKit（接了反而进入健康数据监管）
- 仍需：隐私政策 + 隐私标签（照片含人像、可能有录音 = 个人数据）
- **后端架构无需改造**：苹果不约束前端直连数据库；Supabase 的 anon key + RLS 模型是被接受的范式；敏感操作已有 Edge Functions 兜底
- 上架前用 WebSearch 复核当前隐私/IAP 条款细则（指南随版本更新）

## 九、依赖与排序约束

- 阶段 0 全部先于 Wave 1；其中 **0c 必须先于 dev.2**（浏览屏读字段结构）

- Wave 2（阶段 3）需 Wave 1 + 阶段 2 稳定
- Capacitor 轨道需主线 PWA（v5.1）稳定后启动
