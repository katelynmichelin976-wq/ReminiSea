## 一、初心原点

代码与文档里反复出现、几乎没变过的那句话，就是原点：

> **家庭记忆与学习卡片 PWA —— 看图识字、睡前故事、方言传承，一人制作，家人使用。**
> **妈妈的生日刻在每一张卡片里：MID `1948020901972`。**

从这一句拆出四个原初约定：

| # | 初心 | 含义 |
|---|------|------|
| A | **为妈妈做** | 真实的、唯一最重要的用户是妈妈。一切体验决策的终审者是她。 |
| B | **一人制作，家人使用** | 你是作者，家人是消费者。两种角色，不该混为一谈。 |
| C | **离线即可用** | 卡片导入即用，不依赖网络。云同步是加分项，不是前提。 |
| D | **记忆与陪伴，不是考核** | 「睡前故事、方言传承」是温度词，不是性能指标。 |

整个 776 次提交、若干次大重构，都应该回到这四条来称重。

### 初心的生长（2026-06 确认）

经过大量迭代，你已明确把初心从「为妈妈一人」扩展为一个**新初心**：

> **为更多像我这样的家庭服务，替家属分担。**

这不是 A 的否定，而是 A 的放大——妈妈仍是终审者，但服务对象从「我家」变成「所有照护着像妈妈一样的老人的家庭」。这条新初心是后续一切「向外」工作的总开关，因果链是自觉的：

> 服务更多家庭 → 必须上架（合规 / 多语 / OAuth）→ 用户变多 → 可维护性与运营摆上台面 → 规范化 / 日志可跟踪 / i18n

所以下文 §3.1（复杂度）与 §3.5（公开产品）原先打★的「可疑」判断，**现已确认为有意识的战略选择，不是偏离**——保留原始分析仅作演化留痕。

## 二、演化主线（v4.4 → v5.13）

| 阶段 | 关键节点 | 在做什么 | 对初心 |
|------|---------|---------|--------|
| **接云** | v4.4 Supabase 接入 → v4.7/4.8 统一同步、参数云同步、CDN 媒体 | 从纯本地走向多设备 | 服务 C（离线）+ 扩展（多设备）|
| **定架构** | v5.0 放弃 uni-app，定为 PWA + Capacitor 主线 | 平台路线收敛 | 守住「网址即用」的初心 |
| **内化管理** | v5.0 牌组管理内化进训练 App，废弃独立 deck_manager | 作者工具并入 | 触及 B：作者/使用者边界 |
| **分流双模式** | v5.10 导航重构，standard / advanced 两套 Tab Bar | 把复杂藏给妈妈，留给作者 | **正面回应 B** |
| **软化体验** | v5.11 轻松模式（陪伴）重设计 + 翻卡牌组 | 答错不计惩罚、可重试、自适应节奏 | **正面回应 D** |
| **工程治理** | v5.13.x 命名规范四层、LS_KEYS 注册表、IDB 迁移、日志 ring buffer、诊断配额 | 让代码可维护 | 与 A/D 无关，纯内部 |
| **合规外向** | v5.13.13+ 隐私同意、PP/ToS 多语种、5 语言 i18n | 准备公开发布 | **可能偏离 B**（见 §3.5）|

一句话概括轨迹：**先把「能用、能同步」做扎实 → 再把「对妈妈好」做温柔 → 最后把「能交付给外人」做规范。** 前两段是初心的加深，第三段开始向外生长。

## 三、对照反思（哪里可能偏了）

每条给出：初心怎么想 → 实际怎么走 → 一个诚实的判断。判断仅供参考。

### 3.1 简单 vs 复杂度膨胀 ★最值得自省

- **初心**：CLAUDE.md 第一条编码规则——「最少代码解决问题，不添加未要求的功能，不为单次使用创建抽象」。
- **实际**：单文件 13000+ 行；命名规范写成四层规范文档；localStorage 全量加 `yh:v1:` 前缀走注册表 + helper + 工厂（Phase 1.1→3.1 十余个提交）；IDB 多次 schema 迁移；日志从对象 → event-key 风格 → ring buffer → 按模块配额降采样；17 套单测 + 20+ Playwright 套件。
- **判断（已确认）**：这些治理是新初心的下游——**要服务更多家庭就得上架、能运营、能长期维护**，规范化/日志可跟踪是必要成本，不是洁癖。原先「工程师洁癖」的疑虑就此解除。
- **仍值得的纪律**：方向对，不等于次序对。当用户从「妈妈一人」变成「不认识的家属们」，最高杠杆的事可能不再是 key 前缀或日志配额，而是**新家属第一次打开能不能自己跑起来**（onboarding、错误可自愈、首个牌组从哪来）。把治理精力和「让陌生用户用得起来」的精力放天平上称一称——这是新初心下真正的优先级问题。

### 3.2 一人制作 vs 工具也内化进来了

- **初心**：B——作者和使用者是两个人、两种界面。
- **实际**：v5.0 把牌组管理、制卡内化进训练 App；同时 v5.10 又用 standard/advanced 双模式把它们藏起来。
- **判断**：这一步**没有偏，反而是把初心实现得更干净了**。一个 app 承载两种角色，靠模式分流隔离——妈妈永远看不到「牌组/统计」，你切 advanced 才有。这是对 B 的正解，不是妥协。唯一风险：advanced 模式越长越胖，哪天误触泄漏给妈妈。值得定期自查 `.advanced-only` 的边界。

### 3.3 SRS 严格 vs 轻松模式的「陪伴」

- **初心**：D——记忆与陪伴，不是考核。
- **实际**：早期主线是严格 SM-2（daily_remove_lapses、auto_suspend、learn-ahead）。试用中你加了轻松模式：答错不计 lapse、可重试、按 T 自适应节奏、`history===[1,1,1]` 才算 confident，并配「兜底伪 confident」避免冷场。
- **判断**：这是**初心被试用「校正」的最好例子**。最初照搬 Anki 的算法纪律，是工程师的默认；真用起来才发现妈妈需要的是「陪着她、给她成就感」，于是有了陪伴模式。**这不是偏离，是初心从纸面落到真人身上。** 反过来值得想：normal 模式现在对妈妈还有意义吗？还是说陪伴模式才应该是她的默认、normal 只是你调试用的？

### 3.4 离线即用 vs session 恢复改了 N 次 ★你点名的痛点

- **初心**：C——打开就能用，多设备无感同步。「无感」是关键词。
- **实际**（一条清晰的反复重写线）：
  - `#101` SIGNED_OUT 保护 + 结构化日志
  - `#105` 加 session/sync 事件日志
  - `#165` **重设计**：三级链 → 单次 `getSession()` + 7s 超时
  - 期间反复处理：token 失效、代理/网络断开卡死、backup 损坏、iOS PWA、僵尸 backup 数据清理（`#328`/`#331`）
  - `#524` session 恢复三个独立 bug 再修
- **判断**：**初心没偏，是初心的「野心」撞上了现实的复杂度。** 「无感同步」四个字，背后是 Supabase SDK 异步加载 + token 生命周期 + 离线/弱网 + iOS standalone + 本地 backup 一致性的乘积。每改一次都是因为现实又露出一个没考虑到的角。教训不是「你做错了」，而是：**「无感」是给用户的承诺，对开发者它是最贵的承诺。** 现在它基本稳定（有 `_pw_session_restore.js` 五种失败路径兜底），说明这笔债终于还得差不多了。

### 3.5 家人使用 vs 走向公开产品 ★最可能的真实偏离

- **初心**：B——「一人制作，**家人**使用」。受众是妈妈和家人。
- **实际**：隐私同意 checkbox、PP/ToS 三语全文、5 语言 i18n、App Store 隐私标签待办、OAuth 上架待办、律师复核待办。
- **判断（已确认）**：这**不是偏离，是初心的生长**——从「给妈妈」到「给所有像妈妈一样的老人和他们的家属，替家属分担」。新初心已明确写在 §1，合规/多语/上架都是它的合理产物。原先「mission creep」的疑虑就此解除。
- **生长之后的新责任**：私人 app 的轻装感确实换成了公开产品的责任，这是你主动选的，但责任是真实的——隐私承诺要兑现、协议要经得起律师看（待办里已记）、海外用户的数据合规、出问题要能运营响应。这些不再是「做不做」，而是「做到什么程度才敢上架」。建议把它们从零散待办收成一张**上架就绪清单**，让「能交付」这条线有明确的完成线，而不是无限延展。

### 3.6 温度 vs 去医疗化的自觉

- **初心**：D——温度词，不是临床词。
- **实际**：你专门清理了医疗术语（禁用 AD/患者/认知训练等），主 app 已清，admin/后端另列计划。
- **判断**：**这是初心的一次主动加固，而且加得对。** 把「认知训练」改成「记忆与陪伴」，守住的是妈妈的尊严——她在用一个家人为她做的温柔小程序，不是在做一项病人康复任务。这条不用反思，是做对了的事，值得记住当初为什么这么决定。

## 四、给反思的几个判断题

把上面浓缩成几个你可以直接回答 yes/no 的问题。诚实答完，偏差就清楚了。

1. **每项治理（规范/日志/重构），是「不做就上不了架、运营不下去」，还是「现在不做也行」？** —— 新初心下，前者是必要成本，后者可以排到上架之后。
2. **轻松模式是不是该成为新用户的默认，normal 只是高级档/调试档？** —— 陌生家属第一次用，要的是陪伴感，不是 SM-2 纪律。
3. **session「无感同步」的债基本还清了，下一笔最贵的债是什么？** —— 大概是「陌生家属第一次打开能自助跑起来」（onboarding / 首牌组来源 / 错误自愈）。
4. **新初心已写在 §1。那「上架就绪」的完成线是什么？** —— 列出哪几件事做完就敢让第一个非家人用户进来。
5. **advanced 模式的边界还守得住吗？** —— 用户从家人变成陌生人，误触泄漏的代价更大了。
6. **这 776 次提交里，有几次是「用户能感觉到的」？** —— 这个比例就是初心的体感纯度；服务对象变多后，更要盯着它别掉下去。

## 五、一句话总结（2026-06 定稿）

> 初心没丢，它长大了：从**「为妈妈一人」**长成**「为更多像我这样的家庭，替家属分担」**。
>
> 两条线如今汇成一条：**「对妈妈温柔」**（陪伴模式、去医疗化、双模式藏复杂）是产品的灵魂——它证明你做的东西对真人有效；**「能交付」**（同步、合规、规范、日志、上架）是把这份温柔复制给更多家庭的管道。前者保证值得复制，后者保证能复制。
>
> 既然初心已经说清，往后的反思就不再问「我是不是偏了」，而是问 §4 的优先级题：**在「让更多陌生家属用得起来」这件事上，我现在投的精力，是不是花在了杠杆最大的地方？**
