# 忆海拾光 v5.0 · 腾讯云迁移设计方案

> 设计日期：2026-05-05
> 状态：计划（待实施）
> 前提：v4.x 功能稳定、妈妈试用反馈闭环后启动

---

## 一、背景与动机

### 为什么迁移

| 现状（v4.x） | 问题 |
|-------------|------|
| Supabase (supabase.co) | 被墙，国内用户需 VPN 访问 |
| 邮箱+密码登录 | 不贴合国内手机验证码习惯 |
| 服务器在海外 | 数据合规风险（国内用户数据需存储在国内） |
| 仅 PWA | 无社交传播入口，拉新成本高 |

### v5.0 目标

1. **数据合规** — 所有数据存储在国内服务器
2. **手机验证码登录** — 降低使用门槛
3. **小程序入口** — 微信生态拉新
4. **App 可延续** — 数据平滑迁移到独立 App
5. **PWA 继续可用** — 现有用户无缝过渡

### 技术选型

选 **腾讯云 CloudBase** 而非自建后端（SCF + PostgreSQL + COS）的原因：

| | CloudBase | 自建 SCF + PostgreSQL |
|---|---|---|
| 小程序集成 | **原生 SDK 直连数据库** | 所有操作需写云函数中转 |
| 数据直连 | 小程序端 `db.collection(...).get()` | 必须走 REST API |
| 开发工作量 | 少（小程序端~0 后端代码） | 多（每数据操作写一个云函数） |
| Schema | NoSQL 文档型（需改造） | PostgreSQL 不变 |
| 数据迁移 | 需 ETL（6 表→3 集合） | 可直接导入 |

**选择 CloudBase 的理由：** 小程序接入成本差异巨大。CloudBase 让小程序端直连数据库，对于"基础版小程序拉新 + 正式版 App 留存"的阶段性策略，这是最高效的路径。

## 三、数据库设计：PostgreSQL → CloudBase 集合

### 3.1 现状（v4.x PostgreSQL 6 表）

| 表名 | 用途 | 主要字段 |
|------|------|---------|
| `cards_pool` | 卡片池 | card_id, card_name, deck_name, image_url, audio_url |
| `server_decks` | 牌组索引 | id, name, description, card_count |
| `server_deck_cards` | 牌组-卡片关联 | deck_id, card_id, sort_order |
| `upload_log` | 上传日志 | source_file, deck_name, card_count |
| `sync_trials` | 答题记录 | device_id, card_id, rating, response_time_ms, ... |
| `sync_card_states` | SRS 状态 | device_id, card_id, srs_stage, interval, due_date, ... |

### 3.2 目标（CloudBase 3 集合）

#### 集合 `decks` — 牌组与卡片（合并 3 表）

```json
{
  "_id": "deck_hash8",
  "name": "蔬菜",
  "description": "",
  "cardCount": 20,
  "sourceFile": "蔬菜.yhspack",
  "cards": [
    {
      "cardId": "hash_abc",
      "name": "苹果",
      "imageUrl": "storage/abc_img.jpg",
      "audioUrl": "storage/abc_aud.m4a",
      "sortOrder": 1
    },
    {
      "cardId": "hash_def",
      "name": "香蕉",
      "imageUrl": "storage/def_img.jpg",
      "audioUrl": "storage/def_aud.m4a",
      "sortOrder": 2
    }
  ],
  "createdAt": "2026-05-05T00:00:00Z",
  "updatedAt": "2026-05-05T00:00:00Z"
}
```

**为什么可以内嵌：** 牌组创建后很少修改卡片列表，且下载到本地后整个牌组一次性读取。不需要跨表 Join。最大一个牌组一般 < 200 张卡片，文档大小约 50-100KB，完全在云数据库单文档限制（1MB）内。

#### 集合 `card_states` — SRS 状态（原 sync_card_states）

```json
{
  "_id": "auto",
  "userId": "user_openid_or_uid",
  "stateKey": "deckHash8::cardHash_abc",
  "deckKey": "deckHash8",
  "cardId": "cardHash_abc",
  "srsStage": "review",
  "interval": 86400,
  "easeFactor": 1.3,
  "dueDate": "2026-05-06",
  "dueTs": 1746489600,
  "stepIndex": 0,
  "reviewMode": "T1",
  "lapsesStreak": 0,
  "lapsesTotal": 0,
  "suspended": false,
  "updatedAt": 1746403200
}
```

**索引：** `userId + stateKey` 联合唯一索引，`userId + deckKey` 索引

#### 集合 `trials` — 答题记录（原 sync_trials）

```json
{
  "_id": "auto",
  "userId": "user_openid_or_uid",
  "trialId": "uuid",
  "cardId": "cardHash_abc",
  "deckKey": "deckHash8",
  "sessionId": "session_uuid",
  "questionType": "T1",
  "rating": "good",
  "isCorrect": true,
  "attemptNumber": 1,
  "optionsShown": ["card_a", "card_b", "card_c"],
  "correctOption": "card_a",
  "distractorChosen": null,
  "responseTimeMs": 3500,
  "srsStageBefore": "review",
  "intervalBefore": 86400,
  "easeBefore": 1.3,
  "lapsesStreakBefore": 0,
  "lapsesTotalBefore": 0,
  "reviewModeBefore": "T1",
  "srsStageAfter": "review",
  "intervalAfter": 172800,
  "easeAfter": 1.3,
  "timestamp": 1746403200000,
  "createdAt": "2026-05-05T00:00:00Z"
}
```

**索引：** `userId + timestamp` 索引，`userId + deckKey` 索引

#### 集合 `upload_logs` — 上传日志（原 upload_log，仅 deck_manager 使用）

与原结构基本一致，增加 `userId` 字段。

## 四、迁移步骤

### 第一阶段：环境准备

1. 在腾讯云开通 **CloudBase 环境**
   - 套餐选个人版（19.9元/月）即可，后续按需升级
   - 选择国内节点（北京/上海/广州）
2. 开通 **短信服务**
   - 购买 1 万条套餐包（470元，有效期 2 年）
   - 申请短信签名和正文模板
3. 创建数据库集合
   - `decks`（读写权限：所有用户可读，仅管理员可写）
   - `card_states`（读写权限：仅创建者可读写）
   - `trials`（读写权限：仅创建者可读写）
   - `upload_logs`（读写权限：仅管理员可写）

### 第二阶段：数据迁移

4. 将 Supabase 中存量数据导出（PostgreSQL dump 或 CSV）
5. 编写迁移脚本：6 表 → 3 集合 ETL
   - `cards_pool` + `server_decks` + `server_deck_cards` → `decks`（按 deck_name 分组，cards 内嵌）
   - `sync_card_states` → `card_states`（逐条转换）
   - `sync_trials` → `trials`（逐条转换）
6. 将媒体文件从 Supabase Storage 迁移到 COS

### 第三阶段：云函数开发

7. 编写云函数 `sendSmsCode`
   - 接收手机号
   - 调用腾讯云 SMS 发送 6 位验证码
   - 验证码缓存到 Redis（或云数据库 5 分钟过期）
8. 编写云函数 `login`
   - 接收手机号 + 验证码（或微信 login code）
   - 校验验证码（或调用微信 code2session）
   - 签发自定义 JWT（含 userId）
9. 编写云函数 `importDeck`（可选，替代 deck_manager）

### 第四阶段：PWA 改造

10. 修改 `yihai_v4.7.html`（→ `yihai_v5.0.html`）
    - 删除 Supabase SDK（CDN script）
    - 登录 UI：邮箱+密码 → 手机号+验证码
    - `_sb.from(x).insert/upsert/select` → `cloud.callFunction()`
    - `_sb.storage.download()` → COS 签名 URL / 云存储下载 API

### 第五阶段：小程序开发（后续）

11. 微信开发者工具新建小程序项目
12. `app.js` 初始化 CloudBase
13. 页面复用 PWA 的 Vue/Vanilla 逻辑，渲染层改为 WXML
14. 数据库层直接用 `wx.cloud.database()` 直连

## 五、PWA 端改造明细（yh_v4.7.html → v5.0.html）

### 5.1 删除

| 位置 | 内容 | 理由 |
|------|------|------|
| CDN script | `<script src="supabase-js">` | 不再使用 Supabase |
| 常量 | `SUPABASE_URL`, `SUPABASE_ANON_KEY` | 替换为 CloudBase 配置 |
| 函数 | `restoreCloudSession()` | 替换为 JWT Token 恢复 |
| 函数 | `doCloudLogin()` | 替换为 `doCloudLogin(phone, code)` |
| 函数 | `doCloudLogout()` | 替换为清除本地 Token |

### 5.2 新增

| 位置 | 内容 |
|------|------|
| CDN script | `<script src="https://cdn.imgcache.qq.com/tcb/cloudbase-js-sdk/2.x/cloudbase.js">` |
| 常量 | `CLOUDBASE_ENV_ID`, `CLOUDBASE_REGION` |
| 工具函数 | `requestSmsCode(phone)` — 请求验证码 |
| 工具函数 | `cloudLogin(phone, code)` — 验证码登录 |
| 工具函数 | `cloudRequest(path, data)` — 统一 API 请求（携带 JWT） |

### 5.3 替换

| 原代码 | 替换为 |
|--------|--------|
| `_sb.auth.signInWithPassword(...)` | `cloudLogin(phone, code)` → 获取 JWT |
| `_sb.auth.getSession()` | 读取 localStorage 中的 JWT，检查过期 |
| `_sb.from('sync_trials').insert({...})` | `cloudRequest('/sync/trials', { body })` |
| `_sb.from('sync_card_states').upsert({...})` | `cloudRequest('/sync/states', { body })` |
| `_sb.from('server_decks').select('...')` | `cloudRequest('/decks')` |
| `_sb.from('server_deck_cards').select('...')` | `cloudRequest('/decks/' + deckId)` |
| `_sb.storage.from('ReminiSea').download(url)` | 云存储 `getTempFileURL(url)` 或 COS 签名 URL |

### 5.4 整体改动量

| 模块 | 改动类型 | 预估行数 |
|------|---------|---------|
| 配置与脚本 | 替换 | ~5 行 |
| Auth 登录/登出 | 重写 | ~50 行 |
| 数据同步（sync_trials/states） | 替换 API 调用 | ~60 行 |
| 云端牌组浏览（下载） | 替换 API 调用 | ~80 行 |
| 文件下载 | 替换存储层调用 | ~30 行 |
| UI 表单（登录界面） | 修改 | ~20 行 |
| **合计** | | **~250 行** |

## 六、小程序端设计要点（预留）

### 6.1 技术方案
- 使用 **微信小程序原生框架**（非 uni-app / Taro，减少抽象层）
- CloudBase SDK 原生集成
- 若需跨平台（支付宝/抖音小程序），后续再抽象

### 6.2 页面结构预估

```
pages/
├── index/           # 首页：今日学习卡片
├── decks/           # 牌组列表 / 选择
├── study/           # 答题页（T1 测验）
├── stats/           # 学习统计
├── cloud/           # 云端牌组浏览 + 下载
├── profile/         # 个人中心
└── login/           # 手机号登录（首次）
```

### 6.3 数据互通关键
- 小程序用 `openid` 标识用户
- App/PWA 用手机号标识用户
- CloudBase 云函数中建立 `openid ↔ phone` 映射关系
- 同一用户在小程序上的进度 → App 登录后自动同步

## 八、风险与决策

### 已知风险
1. **腾讯云 SMS 签名审核** — 首次申请短信签名需 1-3 个工作日审核，且签名需与主体资质一致（个人开发者可能受限）
2. **微信小程序审核** — 医疗健康类小程序可能需要额外资质
3. **NoSQL 数据一致性** — 无事务保证，需要应用层处理并发
4. **Supabase 到 CloudBase 的同步过渡期** — 迁移期间双写？迁移工具需要保证数据完整性

### 待决策项
- [ ] 是否保留 Supabase 数据作为只读冷备？
- [ ] 迁移窗口期：停服迁移还是双写过渡？
- [ ] 小程序的 AI 辅助认知训练是否需要单独开发？
- [ ] 是否需要兼容 v4.x 离线导出的 `.yhspack` 文件导入到 CloudBase？
